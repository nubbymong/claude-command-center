Layout System: Which model fits best?

This determines how Side Chat, Diff Viewer, Preview, and Editor panels coexist with the terminal

A

Split Pane (VS Code style)

Drag dividers to split the main area into 2-4 resizable regions. Each region holds one panel (terminal, diff, preview, editor). Panels can be swapped between regions.

Pros

  • Familiar to devs (VS Code, JetBrains)
  • Flexible sizing
  • Panels always visible

Cons

  • Complex drag system
  • Screen real estate splits quickly
  • State persistence is complex
B

Tabbed Panels (Chrome DevTools style)

Main area stays as the terminal. A secondary panel area (right or bottom) has tabs for Diff, Preview, Editor, Side Chat. User toggles which secondary panels are visible and where they dock.

Pros

  • Terminal stays primary (our core value)
  • Much simpler to implement
  • Familiar DevTools pattern
  • Easy state persistence (which tabs open, dock position)

Cons

  • Only one secondary panel visible at a time
  • Less flexible than full split
C

Hybrid: Tabbed + Overlay

Terminal stays primary. Side Chat is always an overlay (slide-in panel, Ctrl+;). Diff/Preview/Editor are tabbed panels that dock right or bottom. Best of both: quick access overlays for chat, persistent panels for content.

Pros

  • Side Chat feels instant (overlay, not a panel)
  • Content panels organized with tabs
  • Terminal stays king
  • Natural UX hierarchy: terminal > panels > overlay

Cons

  • Two different panel systems to build
  • Side Chat overlay covers content

My Recommendation: C (Hybrid)

Side Chat as an overlay is the right UX -- it's a quick, ephemeral conversation that shouldn't compete for permanent screen space. Diff/Preview/Editor as tabbed panels keeps things organized without the complexity of a full grid system. This also matches how Claude Code Desktop does it.
