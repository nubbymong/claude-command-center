## Known bugs / work in flight

### 1. Typing into Claude sessions — supposed to work after `ac6372e`, NOT confirmed by user
Root cause was: `.xterm-helper-textarea` was added to `display: none` multi-selector in `src/renderer/components/terminal/terminalTheme.ts`. That textarea is xterm's keyboard input capture surface; hiding it killed all input.
`ac6372e` removes it from the hide list, applies `caret-color: transparent` separately so the textarea stays focusable.
**Verify on next session:** start dev server, spawn a session, type text, confirm it reaches Claude.

### 2. Light-mode terminal sync feels delayed
User reported: "switching dark/light mode still isn't instantly applying to the terminal". My current implementation:
- `useThemeController` mutates `document.documentElement.dataset.theme`
- `TerminalView` MutationObserver on `<html data-theme>` fires `apply()` which schedules a `requestAnimationFrame` to read CSS vars and call `term.options.theme = ...; term.refresh(0, rows-1)`.
- The MutationObserver pattern is correct, but the user perceives lag. Possible reasons: rAF gives one frame delay (~16ms), `term.refresh` may not fully repaint xterm canvas content. Investigate xterm-internal mechanism for "refresh entire buffer with new theme".

### 3. Light palette possibly still too bright
First draft was paper-white (`#f6f5f3` base). Second draft (current `49c7e50`) is "foggy daylight" `#d6d4d0` warm medium grey. User hasn't confirmed if this is acceptable. If still bright, go even darker — try `#c4c1bd` base.

### 4. "App Dev" → "App" truncation in Claude's TUI status pill
**Not our code.** Claude Code's TUI renders the working-directory basename (or some inferred project name) in the cyan pill at bottom-right of the terminal area. We dropped `--name` flag earlier, so we're not feeding any label. Investigate further only if user persists — would require digging into Claude Code internals or testing alternative workarounds (e.g., setting `name` field in `~/.claude/settings.json`).
